## 2. office-hours 진행 로그 (질문·답변 그대로)

세션은 YC office-hours 방법론(현실을 못 박는 강제 질문 → 전제 검증 → 대안 생성 → 설계 문서)으로 진행 중.
질문은 한 번에 하나씩 던지고 답을 기다리는 방식.

### Q0 — 1차 목표 (모드 결정)
- **질문**: 이 에이전트의 1차 목표는? (우리 조직 실무 도구 / 외부 판매 제품 / 학습·실험 / 미정)
- **답변**: **"우리 조직 실무 도구"**
- **귀결**: `builder / intrapreneurship` 모드 확정. 외부 수요·웨지 검증보다 "우리 조직이 실제로 더 좋은
  과제를 빠르게 찾는 도구를 빠르게 만든다"에 초점.

### Q1 — 현 상태(status quo) (복수 선택)
- **질문**: 지금 이 에이전트 없이 정부과제를 찾을 때, 조직에서 실제로 가장 아픈 지점은?
- **답변 (3개 선택)**:
  1. ✅ **공고 수집 자체가 고통** — IRIS·NTIS·기업마당·K-Startup·부처 홈페이지를 일일이 돌며 모으는
     것 자체가 시간 소모. 새 공고를 놓치기도 함.
  2. ✅ **정보가 흩어짐·놓쳐짐** — 누가 어떤 공고를 봤는지 공유가 안 되고, 마감을 놓치거나 중복 검토.
  3. ✅ **외부 의존/곧비용** — 에이전시·컨설팅에 맡기거나 수수료를 냄. 내재화로 비용·의존을 줄이려 함.
  - ❌ (선택 안 함) "선별·적합성 판단이 고통" → **즉, 매칭/점수화는 현재 가장 아픈 곳이 아님.**
- **귀결(핵심 신호)**: 병목은 **상류(수집·집계·놓침 방지)** 에 있다. 매칭·랭킹은 그 다음 문제.

### Q2 — 가장 좁은 웨지(wedge)
- **질문**: 다음 주에 딱 하나만 제대로 작동해 바로 실무에 쓸 첫 버전이라면?
  (수집+알림 / 프로필→A·B·C 리스트 / 공고문 자동요약 / 검색 가능한 DB)
- **답변**: **"수집+알림 (안 놓치게)"**
- **귀결**: 첫 버전 = 여러 소스의 신규 공고를 한 곳에 모아 **중복 제거 + 마감 임박 표시 + 알림**.

### ⚠️ 발견한 갭 (office-hours의 핵심 통찰)
- 사용자가 처음 그린 그림(요청 3번) = *"회사 프로필을 넣으면 → 매칭된 요약 리스트가 나온다"* = **랭킹/매칭 도구**.
- 그러나 실제로 확인된 통증(Q1) = *"모든 공고를 빠짐없이 한 곳에 모으는 것"* = **수집·집계 도구**.
- 둘은 **다른 물건**이다. 보통 **수집을 먼저 못 박아야 그 위의 매칭이 의미**가 있다.
- → 그래서 1차 웨지를 수집+알림으로 잡고, 매칭/A·B·C 랭킹은 2단계로 분리하기로 함 (Q2에서 사용자도 동의).

## 3. 설계 백본 — `government_project_research_categories.md` 요약

이 파일(repo 루트)은 **기업 관점 정부과제 분류·선별 체계**를 이미 잘 정리해 두었고, 에이전트 로직의
뼈대로 거의 그대로 쓸 수 있다. 핵심만 압축:

**핵심 철학**: "공고를 많이 모으는 것"이 아니라 **"우리 회사가 실제 지원 가능하고 사업적으로 활용 가능한
과제를 빠르게 선별"** 하는 것이 목표. 최종 산출은 **A / B / C / 제외** 우선순위.

**분류 축 (태깅 스키마로 직결)**:
- A. 사업 목적: R&D·기술개발 / 실증·PoC / 사업화·판로 / 창업·스케일업 / 인력·고용 / 수출·해외 / 바우처·컨설팅
- B. 기업 적합성: 스타트업 / 중소 / 중견 / 대기업 / 산학연 협력 필요 / 지역 제한
- C. 과제 규모: 소형(~5천만) / 중소형(5천만~2억) / 중형(2억~10억) / 대형(10억+) / 초대형(수십억+)
- D. 수행 방식: 단독 / 산학연 / 컨소시엄 / 수요연계 / 실증 / 바우처 / 지정공모 / 자유공모
- E. 기술·산업 분야: AI·생성형 / 데이터 / 클라우드·SaaS / 보안 / 제조 / 헬스케어 / 금융 / 로봇·자율주행 / 콘텐츠 / 공공·국방
- F. 실무 리스크: 민간부담금 / 기술료 / 정산 부담 / 실증처 필요 / 컨소시엄 필요 / IP 귀속 / 보안·개인정보 / 마감 임박

**추천 자료조사 프로세스 (에이전트 파이프라인으로 직결)**:
1. 공고 수집 (IRIS / NTIS / 기업마당 / K-Startup / 부처·전문기관 / 지자체·TP / 나라장터)
2. 1차 필터링 (자격·지역·업력·기술분야 불일치 / 마감 임박 / 민간부담금 과도 / 실증처 불가 → 제외)
3. 카테고리 태깅 (위 A~F 축)
4. A/B/C/제외 우선순위 분류
5. A급 과제만 심층 분석 (RFP·평가배점·선정규모·경쟁률·제출서류·예산·기술료·발표평가)
6. A급은 1페이지 기획서부터 (문제정의·정부지원필요성·핵심기술·정량목표·추진체계·사업화계획)

**관리 컬럼(DB 스키마 후보, §11)**: 공고명 / 출처URL / 소관부처 / 전문기관 / 사업목적 / 기술분야 /
신청대상 / 우리회사 역할(주관·공동·수요·위탁) / 지원금 규모 / 민간부담금 / 수행기간 / 접수마감 /
제출난이도 / 선정가능성 / 사업적합도 / 필요파트너 / 핵심제출서류 / 리스크 / 우선순위(A·B·C·제외) / 다음 액션.

**점수화 모델(§9)**: 긍정 점수(기술적합도·자격충족·지원금규모·선정가능성·사업화효과·마감여유) vs
리스크 점수(작성난이도·행정정산부담·컨소시엄확보·민간부담금). 총점 높아도 리스크 점수 높으면 우선순위 하향.

**운영(§15)**: 주 1회 신규 공고 수집 · 월 1회 A/B/C 재분류 · 분기 1회 기술전략 매칭 검토.

> **시사점**: 분류·점수화·DB 스키마·프로세스는 이미 설계 완료 상태. 에이전트가 **새로 풀어야 할 진짜
> 난제는 "1단계(수집)를 어떻게 빠짐없이·자동으로·합법적으로 하느냐"** 이다 → 이게 다음 조사 대상.

## 4. 핵심 미해결 질문 (다음 작업의 1순위)

> **설계를 가르는 단 하나의 미지수: 한국 정부과제 데이터 소스를 실제로 어떻게 수집하는가?**

각 소스가 ① 공개 API가 있는지, ② 없으면 스크래핑이 필요한지, ③ 스크래핑이 법적/약관상 가능한지에 따라
아키텍처가 완전히 달라진다. 추측으로 넘기지 말고 **소스별로 사실 확인**해야 함.

### 4.1 조사 대상 소스 + 현재 가설 (★ 반드시 검증 필요)

| 소스 | 무엇 | 접근 가설 (미검증) | 확인할 것 |
|---|---|---|---|
| **공공데이터포털 (data.go.kr)** | 정부 공개 API 허브 | 아래 여러 소스의 Open API가 여기 모여 있을 가능성 높음. 인증키 발급 필요 | 어떤 과제·공고 API가 실제 등록돼 있나, 필드·갱신주기·호출제한 |
| **기업마당 (bizinfo.go.kr)** | 중기부 지원사업 공고 통합 | 공공데이터포털에 지원사업 공고 Open API 존재 가능성 | API 엔드포인트·필드(공고명·기관·마감·대상)·인증 방식 |
| **K-Startup (k-startup.go.kr)** | 창업지원사업 공고 | 공공데이터포털에 사업공고 Open API 존재 가능성 | 동일 |
| **NTIS (ntis.go.kr)** | 국가R&D 정보(과제·공고·성과) | Open API 제공 이력 있음(별도 신청·승인 필요할 수 있음) | API 신청 절차, 공고 데이터 포함 여부, 승인 난이도 |
| **IRIS (iris.go.kr)** | 범부처통합연구지원(공고·접수) | 공개 API 여부 불확실. 스크래핑 필요 가능성 | API 공개 여부, robots/약관, 로그인 게이트 여부 |
| **나라장터 G2B (조달청)** | 입찰·용역 공고 | 공공데이터포털에 입찰공고 Open API 존재 | 정부과제와 관련된 용역/실증 공고 범위 |
| **부처·전문기관** (IITP·KEIT·KIAT·TIPA·NIPA·창업진흥원·KHIDI·KAIA·KEITI 등) | 개별 공고 페이지 | 일부 RSS, 대부분 HTML 스크래핑 | 기관별 RSS 유무, 페이지 구조 안정성 |
| **지자체·테크노파크(TP)** | 지역 공고 | 표준 없음, 개별 스크래핑 | 우선순위 낮음(롱테일). 1차 범위 제외 후보 |

### 4.2 조사 시 반드시 함께 볼 것
- **스크래핑 합법성**: 공공저작물 이용약관, robots.txt, 각 포털 이용약관(상업적/자동수집 제한 조항).
  정부 공공데이터는 대체로 개방적이나 사이트별 약관 확인 필수. (무출처 단정 금지 — 출처 명시)
- **데이터 정규화 난제**: 소스마다 필드명·날짜형식·기관명 표기·금액 단위가 제각각 → 통합 스키마(§3 컬럼)로
  매핑하는 정규화 레이어가 핵심. 중복 제거 키(공고명+기관+마감 조합 등) 설계 필요.
- **갱신·증분 수집**: 전체 재수집 vs 신규만(증분). 마감 임박·신규 공고 알림을 위한 diff 로직.
- **공고문 원문(PDF/HWP) 처리**: 한국 공고는 HWP(한글) 첨부가 많음 → HWP 파싱이 별도 난제.

## 5. 잠정 아키텍처 스케치 (현재까지의 생각, 미확정)

웨지(수집+알림)를 기준으로 한 잠정 파이프라인. **Phase 4에서 정식 대안 2~3개로 구체화 예정.**

```text
[1] 수집기(Collectors)        소스별 어댑터(API 우선, 없으면 스크래퍼). data.go.kr 키 기반 우선.
        │
[2] 정규화(Normalizer)        통합 스키마(§3 컬럼)로 매핑 + 기관명/금액/날짜 정규화
        │
[3] 중복제거·증분(Dedup/Diff) 공고 고유키로 중복 제거, 신규/변경 감지
        │
[4] 저장(Store)               공고 DB (노션 / 구글시트 / SQLite / 웹 중 택1 — 미정)
        │
[5] 알림(Notify)              신규·마감임박 공고 알림 (이메일/슬랙/카톡 등 — 미정)
        ▼
   ───── (여기까지가 1차 웨지) ─────
        │
[6] 1차 필터(Filter)          자격·지역·업력·분야 불일치 자동 제외
        │
[7] 태깅·점수화(Tag/Score)    A~F 축 태깅 + §9 점수화 모델
        │
[8] 매칭·랭킹(Match/Rank)     회사·연구실 프로필 입력 → A/B/C/제외 랭킹  ← 사용자 원래 요청(요청3)
        │
[9] 심층요약(Summarize)       A급 공고문/RFP 자동 요약 + 체크리스트
```

**열린 설계 결정(아직 안 정함)**:
- 저장소: 노션 vs 구글시트 vs SQLite/로컬 vs 경량 웹앱 (사용자 조직의 기존 도구에 맞춰야 함 → 물어볼 것)
- 알림 채널: 이메일 / 슬랙 / 카카오 / 노션 알림
- 실행 형태: 수동 실행 스크립트 vs 스케줄러(주 1회 자동) vs 상시 서비스
- "에이전트" 정도: 단순 ETL 파이프라인이냐, LLM이 태깅·요약·매칭을 하는 진짜 에이전트냐
- HWP 첨부 파싱 포함 여부 (1차 범위 제외 후보)
